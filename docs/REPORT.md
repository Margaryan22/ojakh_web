# Веб-приложение для автоматизации продаж домашних полуфабрикатов и кондитерских изделий «Ojakh»

---

## ВВЕДЕНИЕ

### Обоснование актуальности

В последние годы в России наблюдается устойчивый рост сегмента домашнего производства продуктов питания. Частные кулинары, производящие домашние полуфабрикаты, торты, десерты и другие изделия ручного приготовления, всё активнее выходят на рынок. При этом большинство из них использует примитивные инструменты взаимодействия с покупателями: мессенджеры (WhatsApp, Telegram), социальные сети и ручной учёт заказов в таблицах или тетрадях.

Подобный подход порождает ряд системных проблем: потеря заказов, ошибки в расчётах, невозможность контролировать загрузку производства, отсутствие истории покупок у клиентов. По данным исследований рынка e-commerce, предприниматели, переходящие на автоматизированные онлайн-платформы, увеличивают выручку в среднем на 30–50% уже в первые полгода, за счёт снижения потерь заказов и повышения удобства для покупателей.

Разработка специализированного веб-приложения, учитывающего специфику малого домашнего производства (ограниченные производственные мощности, необходимость регулировать число заказов на конкретные даты, управление слотами для сложных изделий — тортов), является актуальной практической задачей.

### Постановка проблемы

**Как есть (AS-IS):** Домашний производитель принимает заказы через личную переписку в мессенджерах. Клиент пишет сообщение, уточняет состав, сроки. Производитель вручную фиксирует заказ, согласовывает дату доставки и стоимость. Оплата производится переводом на карту. Контроль загрузки на конкретную дату отсутствует — возможна ситуация, когда принято больше заказов, чем производитель физически может выполнить.

**Как должно быть (TO-BE):** Клиент самостоятельно просматривает каталог продукции на сайте, добавляет товары в корзину, выбирает дату доставки из доступных слотов, оформляет и оплачивает заказ. Производитель видит все заказы в административной панели, управляет статусами, контролирует загрузку производства по дням. Клиент получает уведомления об изменении статуса своего заказа.

### Объект и предмет работы

**Объект исследования** — процесс организации продаж и управления заказами в малом домашнем производстве продуктов питания.

**Предмет исследования** — веб-приложение как средство автоматизации процессов приёма заказов, управления каталогом продукции, контроля производственной загрузки и уведомления клиентов.

### Цель и задачи работы

**Цель работы** — разработка и внедрение веб-приложения для автоматизации процесса продаж домашних полуфабрикатов и кондитерских изделий, обеспечивающего удобное взаимодействие клиента с производителем.

**Задачи работы:**

## ГЛАВА 1. ПРЕДПРОЕКТНЫЙ АНАЛИЗ

### 1.1 Актуальность темы

Рынок домашней еды в России демонстрирует значительный рост. По оценкам аналитиков, ежегодный оборот частных производителей домашней еды превышает несколько миллиардов рублей. Пандемия COVID-19 стала катализатором развития этого сегмента: работа на дому нормализовалась, появился запрос на «домашнее», натуральное питание.

Существующие крупные платформы — Wildberries, Ozon, Яндекс.Маркет — не подходят для домашних производителей по ряду причин:
- Высокие комиссии и требования к сертификации продукции.
- Отсутствие механизма управления производственной загрузкой (невозможно ограничить количество заказов тортов на конкретный день).
- Стандартизированная логистика, не учитывающая специфику скоропортящихся изделий.
- Отсутствие персонального взаимодействия с клиентом.

Разработка собственного специализированного веб-приложения позволяет устранить перечисленные ограничения и обеспечить производителю полный контроль над бизнес-процессами.

### 1.2 Понятийный аппарат

**Веб-приложение** — программное обеспечение, работающее на веб-сервере и доступное пользователю через браузер по протоколу HTTP/HTTPS. В отличие от традиционных сайтов, веб-приложения обеспечивают интерактивное взаимодействие пользователя с данными в реальном времени.

**REST API (Representational State Transfer Application Programming Interface)** — архитектурный стиль взаимодействия компонентов распределённого приложения в компьютерной сети, при котором взаимодействие осуществляется через стандартные HTTP-методы (GET, POST, PUT, PATCH, DELETE).

**SPA (Single Page Application)** — одностраничное веб-приложение, в котором весь необходимый код (HTML, JavaScript, CSS) загружается единожды при первом обращении, а последующие обновления страницы выполняются динамически без полной перезагрузки.

**JWT (JSON Web Token)** — открытый стандарт для создания токенов аутентификации, используемый для безопасной передачи информации между сторонами в формате JSON.

**ORM (Object-Relational Mapping)** — технология программирования, позволяющая работать с реляционной базой данных через объектно-ориентированный интерфейс, без написания SQL-запросов вручную.

**Docker** — платформа контейнеризации, позволяющая упаковать приложение и все его зависимости в изолированный контейнер, обеспечивая воспроизводимость среды выполнения.

**Slot-based scheduling** — подход к управлению производственными мощностями, при котором каждый временной слот (дата) имеет ограниченное количество доступных позиций. В контексте данного проекта — ограничение на число заказов и тортов на конкретную дату доставки.

### 1.3 Критический анализ текущего состояния проблемы

Домашний производитель «Ojakh» до внедрения системы использовал следующие инструменты:
- Приём заказов: личная переписка в WhatsApp/Instagram Direct.
- Учёт заказов: таблица Google Sheets.
- Согласование дат: в ручном режиме через переписку.
- Оплата: перевод по номеру карты.
- Уведомления клиентов: ручные сообщения в чате.

**Выявленные проблемы:**
- Среднее время обработки одного заказа — 15–20 минут переписки.
- ~10–15% заказов теряется (забыты в переписке, не внесены в таблицу).
- Отсутствует контроль загрузки: были случаи, когда принималось 4–5 тортов на одну дату при реальной мощности 2 торта/день.
- Клиент не имеет истории своих заказов.
- Производитель тратит 1–2 часа ежедневно на административную работу вместо производства.

### 1.4 Характеристика предметной области

Предметная область — малое домашнее производство продуктов питания. Ассортимент продукции:
- **Полуфабрикаты** — замороженные или охлаждённые готовые к приготовлению изделия (пельмени, манты, самса и др.). Единица измерения: штуки или килограммы. Шаг заказа может быть дробным.
- **Торты** — изделия на заказ, требующие 1–2 суток на производство. Являются лимитированной позицией (не более 2 тортов на дату).
- **Десерты** — кондитерские изделия меньшего объёма (пирожные, кексы и т.д.).

**Информационные потребности пользователей:**

| Роль | Информационная потребность |
|------|---------------------------|
| Клиент | Просмотр каталога, наличие товаров, стоимость, доступные даты доставки, статус заказа |
| Администратор | Список заказов по дате/статусу, загрузка производства, управление статусами, управление продуктами |

### 1.5 Описание модели деятельности «как есть» (AS-IS)

Процесс приёма заказа AS-IS включает следующие шаги:
1. Клиент пишет сообщение в мессенджер с составом желаемого заказа.
2. Производитель просматривает сообщение (задержка до нескольких часов).
3. Производитель уточняет детали заказа (количество, дата, адрес).
4. Производитель проверяет свободные даты в таблице.
5. Производитель сообщает итоговую стоимость.
6. Клиент подтверждает и оплачивает.
7. Производитель вносит заказ в таблицу вручную.
8. В день доставки производитель уведомляет клиента через мессенджер.

Среднее число шагов взаимодействия: 8–12. Время полного цикла: 1–3 часа.

### 1.6 Анализ нормативно-правовых документов

При разработке системы учитывались следующие нормативные акты:
- **Федеральный закон № 152-ФЗ «О персональных данных»** — требования к хранению и защите персональных данных пользователей (email, имя, телефон, адрес доставки).
- **Федеральный закон № 149-ФЗ «Об информации, информационных технологиях и о защите информации»** — требования к информационным системам.
- **ГОСТ Р 34.12-2015** — стандарт шифрования данных. Реализован через SSL/TLS (Let's Encrypt).
- **Закон № 2300-1 «О защите прав потребителей»** — обязательное отображение цены, состава и условий доставки.
- **WCAG 2.1** — рекомендации по доступности веб-контента (достигнута частичная совместимость: контрастность текста, адаптивная вёрстка).

3. Разработать клиентское SPA: каталог, корзина, оформление заказа, история заказов, профиль, избранное, административная панель.
4. Реализовать механизм проверки и управления слотами доставки.
5. Реализовать ролевую модель (клиент / администратор).
6. Настроить производственную инфраструктуру: Docker, Nginx, SSL.
7. Выполнить развёртывание на VPS-сервере.

### 1.9 Исходные параметры объекта проектирования

- Ожидаемая нагрузка: до 50 уникальных пользователей в сутки, до 20 заказов в день.
- Максимальное число тортов в день: 2 (настраивается через модель DailyLimit).
- Максимальное число заказов в день: 15 (настраивается).
- Окно планирования: 2–15 дней вперёд.
- Валюта расчётов: рубли (хранятся в копейках для исключения ошибок округления).
- Требования к доступности: 99% (VPS с автоматическим перезапуском контейнеров).

Проведённый сравнительный анализ аналогов показал, что существующие решения не охватывают специфические потребности домашнего производителя. Сформулированы задачи на проектирование и определены исходные параметры системы.

### 2.2 Функциональная модель «как должно быть» (TO-BE)

#### 2.2.1 Портреты целевых пользователей (Персоны)

**Персона 1: Клиент — Алина, 32 года**
- Работает менеджером, активный пользователь смартфона.
- Заказывает полуфабрикаты 2–3 раза в месяц для семейного ужина.
- Ценит удобство: хочет быстро выбрать товар, оплатить и получить уведомление.
- Болевые точки: не хочет вести переписку, нужна ясность с датами и стоимостью доставки.

**Персона 2: Постоянный клиент — Дмитрий, 45 лет**
- Заказывает торты на праздники (2–4 раза в год).
- Планирует заказ заранее, важна возможность выбрать конкретную дату.
- Нуждается в подтверждении заказа и уведомлениях о готовности.

**Персона 3: Администратор — производитель Гульнора**
- Готовит продукцию самостоятельно, ограниченные мощности.
- Нуждается в обзоре всех заказов на ближайшие дни.
- Важно не перегрузить производство (особенно по тортам).
- Хочет управлять статусами заказов и видеть загрузку по дням.

#### 2.2.3 Сценарии использования (User Stories)

**Сценарий 1: Оформление заказа с доставкой**
1. Клиент открывает сайт → система перенаправляет на каталог.
2. Клиент просматривает категории (полуфабрикаты, торты, десерты).
3. Клиент нажимает «Добавить» → выбирает количество в диалоге.
4. Клиент переходит в корзину → видит позиции и итоговую сумму.
5. Клиент выбирает «Доставка» → система предлагает календарь доступных дат.
6. Система проверяет: если в корзине есть торт — показывает только даты с доступными слотами для тортов.
7. Клиент выбирает дату, вводит адрес.
8. Клиент нажимает «Оформить» → система создаёт заказ.
9. Клиент оплачивает заказ.
10. Система отправляет уведомление: «Заказ #N принят».

**Сценарий 2: Управление заказом администратором**
1. Администратор открывает панель → видит список заказов на сегодня/завтра.
2. Администратор начинает приготовление → нажимает «Начать приготовление» → статус меняется на «Готовится».
3. Клиент получает уведомление об изменении статуса.
4. Администратор завершает приготовление → нажимает «Готово» → статус «Готов».
5. Клиент получает уведомление «Заказ готов, ожидает доставки».

### 2.5 Обоснование выбранных методов и средств разработки

#### Frontend: Next.js 16 + React 19

Next.js выбран как наиболее зрелый React-фреймворк, обеспечивающий:
- App Router для файловой маршрутизации.
- Server Components для оптимизации загрузки.
- Встроенную оптимизацию изображений.
- Простоту развёртывания в Docker.

Альтернативы: Vite + React (отсутствие SSR), Vue 3 (меньшая экосистема), Angular (избыточная сложность для данного масштаба).

#### Backend: NestJS + Fastify

NestJS выбран благодаря:
- Модульной архитектуре, облегчающей разделение ответственности.
- Встроенной поддержке TypeScript и декораторов.
- Интеграции со Swagger для автодокументирования API.
- Fastify как адаптер платформы (в 2–3 раза быстрее Express в тестах throughput).

#### База данных: PostgreSQL + Prisma

PostgreSQL обеспечивает:
- ACID-транзакции (критично для операций с заказами и слотами).
- Индексирование по `deliveryDate` и `status` для эффективной выборки.
- Надёжность и масштабируемость.

Prisma ORM выбран за:
- Type-safe запросы (ошибки на этапе компиляции, а не выполнения).
- Встроенные миграции схемы.
- Генерацию клиента из декларативной схемы.

#### Управление состоянием: Zustand

Легковесная альтернатива Redux. Используется для хранения данных аутентифицированного пользователя. React Query (TanStack Query) применяется для серверного состояния: кэширование ответов API, автоматическая инвалидация кэша.

#### Инфраструктура: Docker + Nginx + Let's Encrypt

Docker Compose оркестрирует 5 сервисов: postgres, backend, frontend, nginx, certbot. Nginx выступает reverse proxy с SSL-терминацией и автоматическим обновлением сертификатов через Certbot.

#### 2.8.2 Ключевые технические решения

**Хранение корзины:** Корзина хранится в БД как JSON-массив (модель `Cart`). Это позволяет синхронизировать корзину между устройствами авторизованного пользователя, избегая сложной реляционной таблицы `CartItems`.

**Слотовая система доставки:** Модель `DailyLimit` хранит максимальные ограничения по дате. При оформлении заказа сервис `DeliveryService` подсчитывает активные заказы (`new`, `paid`, `preparing`, `ready`) на запрошенную дату. Отдельно считается количество позиций категории «торты» по всем активным заказам.

**Снимок заказа (snapshot):** Поле `Order.items` хранит JSON-массив позиций на момент создания заказа. Это гарантирует, что изменение цены или удаление продукта из каталога не повлияет на историю заказов.

**Хранение цен:** Все цены хранятся в копейках (Integer) для исключения ошибок округления при арифметике с плавающей точкой.

**Уведомления:** При изменении статуса заказа создаётся запись в таблице `Notification`. На клиенте история заказов опрашивается каждые 30 секунд (`refetchInterval: 30000`), при изменении статуса показывается toast-уведомление.

Стоимость разработки (при ставке разработчика 2 000 руб/час):
- Стоимость труда: 250 × 2 000 = **500 000 рублей**

Инфраструктурные расходы (в месяц):
- VPS-сервер (2 CPU, 4 GB RAM): ~1 500 руб/мес
- Домен: ~150 руб/год
- SSL-сертификат: **бесплатно** (Let's Encrypt)

**Срок окупаемости:** При снижении потерь заказов на 15% и увеличении количества заказов на 30% (реалистичная оценка для первых 6 месяцев), система окупается в течение 6–8 месяцев эксплуатации.

**Экономический эффект:**
- Экономия времени администратора: ~1,5 часа в день × 250 рабочих дней = 375 часов/год.
- При стоимости времени 500 руб/час: 187 500 руб/год экономии.
- Снижение потерь заказов (~5 заказов/мес × 600 руб средний чек): 36 000 руб/год.
- Итого годовой эффект: ~223 500 руб/год.

6. Настроена production-инфраструктура: Docker Compose, Nginx, SSL/TLS. ✓
7. Проведена апробация системы в реальных условиях. ✓

**Практическая значимость:** Система внедрена и используется действующим домашним производителем. Время обработки одного заказа сократилось в 5–7 раз. Исключены потери заказов и перегрузка производства.

**Теоретическая значимость:** Разработанный подход к slot-based scheduling для малого производства, реализованный через модель `DailyLimit` и сервис проверки доступности, может быть применён в аналогичных задачах для других типов малого бизнеса (барбершопы, мастерские, фотостудии и т.д.).

**Перспективы и направления продолжения работы:**
1. Интеграция с реальным платёжным шлюзом (ЮKassa, Тинькофф Касса).
2. Реализация push-уведомлений через Telegram Bot API.
3. Добавление программы лояльности (бонусные баллы за заказы).

5. Аналитический дашборд для производителя (статистика продаж, популярные позиции, выручка по периодам).
